Implementation Report

TinyINR Website Upgrade

This note documents the Next.js website upgrade that ports the provided reference design into the existing TinyINR web app while keeping the App Router architecture.

Goal

The goal was to replace the default create-next-app screen with a polished project website inspired by the reference React app. The final result keeps Next.js, uses TinyINR-specific branding, and adds a blog style homepage plus dynamic article pages.

Framework Decision

The reference implementation was a Vite app using react-router-dom. The TinyINR project already uses Next.js, so the implementation keeps the Next.js App Router and translates route behavior into file-system routes. 

web/app/page.tsx
web/app/post/[slug]/page.tsx

Links use next/link instead of React Router. The dynamic post page uses generateStaticParams so all known posts are prerendered.

Files Updated

  • web/app/page.tsx now renders the TinyINR homepage with a vector-field hero, featured article, and latest article grid.
  • web/app/globals.css now defines the dark theme tokens, prose styling, code styling, selection colors, and local font stacks.
  • web/app/layout.tsx now uses TinyINR metadata and avoids runtime Google font fetching so builds work in restricted environments.
  • web/package.json and web/package-lock.json now include lucide-react and framer-motion.

Design Translation

The reference design language was preserved: dark background, teal accents, card-based article previews, animated vector field, mono metadata, and disclosure blocks for mathematical detail.

The content was adapted from generic FlowMatch branding into TinyINR: coordinate-value data, Fourier embeddings, Tensor core, CUDA kernels, and flow matching are now first-class concepts in the copy.

Next.js Adaptations

  • Client-only behavior is isolated to components that need browser APIs: canvas drawing, disclosure state, and Framer Motion animations.
  • Static post data is imported directly from web/data/posts.ts.
  • Dynamic post pages are statically generated with generateStaticParams.
  • The build no longer depends on fetching Google fonts through next/font/google, which avoids network failures.
