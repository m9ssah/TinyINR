TinyINR Week 3

Tensor Core Implementation Notes

This note explains how to build the CPU-first Tensor module and why it matters for TinyINR's larger INRFlow direction: coordinate-value generative modeling with later GPU acceleration.

The Big Picture

TinyINR works with irregular coordinate-value data. For an image, each training item can be represented as coordinates and values: 

coordinates: [B, H * W, 2]  // x, y positions
values:      [B, H * W, 3]  // RGB values

Later, those coordinates are embedded, sampled, attended to, and used in flow matching. All of those steps need the same basic object: a reliable multidimensional float array. That object is the Tensor.

Your Tensor is the floor. CoordinateBatch, FourierEmbedding, attention, adaptive sampling, and CUDA kernels all depend on a stable memory layout and shape convention.

What A Tensor Really Is

In this project, a Tensor should stay intentionally small. It is not PyTorch. It is a CPU float32 container with shape metadata and a few operations.

Data Buffer

A flat std::vector<float> owns the memory. This gives safe RAII allocation and cleanup without manual new or delete.

Shape

A std::vector<int64_t> describes dimensions, such as [batch, points, channels].

Strides

Strides map multidimensional indices into the flat data buffer. For row-major layout, the last dimension has stride 1.

Row-Major Layout

A shape of [2, 3] looks like a matrix: 

[[a, b, c],
 [d, e, f]]

But in memory it is stored contiguously: 

[a, b, c, d, e, f]

For shape [2, 3], the strides are [3, 1]. The offset formula is: 

offset = i * strides[0] + j * strides[1]

So x.at({1, 2}) points to data[5].

Minimum API

Start with the API that the rest of TinyINR needs: 

class Tensor {
public:
    explicit Tensor(const std::vector<int64_t>& shape);

    static Tensor zeros(const std::vector<int64_t>& shape);
    static Tensor randn(const std::vector<int64_t>& shape);

    float* data();
    const float* data() const;

    const std::vector<int64_t>& shape() const;
    const std::vector<int64_t>& strides() const;

    int64_t ndim() const;
    int64_t numel() const;

    float& at(const std::vector<int64_t>& indices);
    const float& at(const std::vector<int64_t>& indices) const;

    Tensor reshape(const std::vector<int64_t>& new_shape) const;
    Tensor row(int64_t row_index) const;

    Tensor add(const Tensor& other) const;
    Tensor multiply(const Tensor& other) const;
    Tensor matmul(const Tensor& other) const;
};
Keep the first version narrow. Do not add autograd, arbitrary dtypes, broad broadcasting, GPU memory, or fancy views yet.

Implementation Order

  1. Constructor, shape storage, and numel().
  2. Stride calculation for contiguous row-major layout.
  3. Safe indexing with bounds checks through at(...).
  4. zeros(...) and deterministic data access tests.
  5. Elementwise add and multiply for matching shapes.
  6. reshape that preserves the number of elements.
  7. 2D matmul as the correctness baseline.
  8. randn once the deterministic path is already tested.

Core Checks

Feature What To Verify Why It Matters
numel() Product of all shape dimensions. Every allocation, reshape, and loop depends on this count.
strides() [B, N, D] becomes [N * D, D, 1]. Coordinate and embedding tensors need predictable layout.
at(...) Correct values and out-of-bounds errors. Early safety prevents silent math bugs.
add / multiply Same-shape elementwise math. Needed for biasing, scaling, masks, and feature transforms.
matmul [M, K] @ [K, N] = [M, N]. Baseline for linear layers and later attention operations.

How This Supports INRFlow

INRFlow is about learning transformations over coordinate-value representations. The Tensor module does not implement flow matching directly, but it makes the later pieces possible.

Coordinate Data

CoordinateBatch will store coordinates and values as tensors with shared shape conventions.

Fourier Features

FourierEmbedding will transform coordinates into higher-dimensional features using tensor reads and writes.

Sampling

Adaptive coordinate sampling will gather subsets of points, compare values, and build new tensors for training.

Attention

Latent-to-coordinate attention eventually relies on matrix multiplication and stable batched shapes.

Shape Convention To Protect

Standardize these shapes now and keep them consistent: 

coordinates: [batch_size, num_points, coordinate_dim]
values:      [batch_size, num_points, value_dim]
embedding:   [batch_size, num_points, embedding_dim]

For image data: 

coordinates: [B, H * W, 2]
RGB values:  [B, H * W, 3]

This convention carries cleanly into Fourier features, attention, adaptive sampling, and CUDA kernels.

First Success Target

The first meaningful Tensor milestone is this: 

Tensor x({batch, dims});
Tensor weights({dims, hidden});
Tensor bias({batch, hidden});

Tensor y = x.matmul(weights);
Tensor z = y.add(bias);

If this works with tests, the rest of TinyINR has something real to build on.
